# Anchor Links

## Status
Accepted

## Goal
Make in-document anchor links (e.g. `[Section](#section)`) navigate to the
target heading, so that tables of contents and cross-references within a
document work as expected.

## Context
- Mindex renders markdown via pulldown-cmark. Headings currently render as
  plain `<h1>`, `<h2>`, etc. without `id` attributes.
- Fragment links (`#section`) are already preserved correctly during link
  rewriting. The missing piece is heading IDs for the browser to scroll to.
- Markdown documents commonly include tables of contents with GitHub-flavored
  anchor links (e.g. `[Proposed Design](#2-proposed-design)`). These are
  generated by GitHub, editors, and LLMs using a well-known slug algorithm.
- The pulldown-cmark event loop already intercepts heading events (indirectly,
  since it intercepts code blocks, math, etc.), so the interception point
  exists.

## Options explored

### Option A: Server-side heading IDs in the event loop
**Approach**
- In `render_document_html`, intercept `Start(Tag::Heading { .. })` and
  `End(TagEnd::Heading(..))` events.
- Collect the text content between Start and End to build the heading text.
- Generate a GitHub-flavored slug from the text (lowercase, replace spaces
  with `-`, strip non-alphanumeric characters except `-`).
- Emit the opening `<hN>` tag manually with an `id` attribute, then let the
  inner content and closing tag flow through normally.
- Track seen slugs to append `-1`, `-2`, etc. for duplicates.

**Pros**
- No new dependencies; slug generation is ~10 lines of string manipulation.
- IDs are present in the initial HTML response, so anchor links work
  immediately with zero JS and no FOUC/flash-of-unlinked-content.
- Works with JS disabled.
- Follows the existing interception pattern (mermaid, abc, math).
- Server-rendered heading IDs are also useful if we ever add a table of
  contents generator or cross-document anchor links.

**Cons**
- Requires buffering heading content to compute the slug before emitting the
  opening tag. This means heading events must be collected and re-emitted,
  adding some complexity to the event loop.
- The slug algorithm must match what authors use in their markdown. If an
  author uses a non-GitHub convention, their links won't resolve. (This is
  inherent to any slug-based approach.)

**Complexity:** Low. ~40-60 lines in `documents.rs`, no new dependencies, no
template changes.

### Option B: Client-side via JavaScript
**Approach**
- Add a small JS module that walks all heading elements on page load,
  generates slugs, and sets `id` attributes.
- Load unconditionally or behind a flag.

**Pros**
- Zero Rust code changes beyond (optionally) a template include.
- Can be swapped or tweaked without recompilation.

**Cons**
- Requires JS. Anchor links in the URL bar on page load (`/doc/foo.md#section`)
  won't work because the IDs don't exist until JS runs, causing a race
  condition with the browser's native fragment scroll. This is the primary
  failure mode: the most common way to hit an anchor link is via a direct URL
  or a click that triggers a same-page navigation, both of which the browser
  handles before JS executes.
- Adds a JS file to serve and maintain.
- Unlike highlight.js or mermaid (which enhance visual presentation), heading
  IDs are structural and affect navigation correctness. Client-side generation
  means degraded behavior without JS.
- Diverges from the mermaid/abc/highlight.js pattern: those enhance rendering;
  this fixes navigation. Server-side is the natural home for structural HTML
  attributes.

**Complexity:** Low. ~20 lines of JS, but with the race condition caveat.

### Option C: pulldown-cmark `heading_attributes` option
**Approach**
- Enable pulldown-cmark's `ENABLE_HEADING_ATTRIBUTES` parser option, which
  lets authors write `## Heading {#custom-id}` to manually specify IDs.
- pulldown-cmark passes the `id` through in the `HeadingLevel` tag attributes.

**Pros**
- Uses pulldown-cmark's built-in support, minimal code.
- Authors have full control over IDs.

**Cons**
- Requires authors to manually annotate every heading. Existing documents with
  GitHub-flavored ToCs (`[Link](#slug)`) would not work without editing every
  heading.
- Not compatible with standard markdown. Documents become less portable.
- Doesn't solve the common case: auto-generated ToCs from GitHub, editors, or
  LLMs that assume GitHub-style slugs.

**Complexity:** Minimal code, but high author burden and poor compatibility
with existing documents.

## Recommendation
**Option A (server-side heading IDs)**, for the following reasons:

1. **Correctness.** Heading IDs are structural, not cosmetic. The browser's
   native fragment navigation (`#section` in URL, click-to-scroll) depends on
   IDs being present in the initial HTML. Server-side generation is the only
   approach that handles all navigation scenarios correctly, including direct
   URL access.
2. **Zero dependencies.** The GitHub-flavored slug algorithm is simple string
   manipulation. No crate needed.
3. **Compatibility.** GitHub-flavored slugs are the de facto standard. ToCs
   generated by GitHub, VS Code, and LLMs all use this convention. Matching it
   means existing documents work without modification.
4. **Minimal code.** The change is contained entirely within `documents.rs`
   and follows the established event-interception pattern. No template, asset,
   or dependency changes.

## Slug algorithm

GitHub-flavored heading slug generation:
1. Convert to lowercase.
2. Strip all characters except alphanumerics, spaces, and hyphens.
3. Replace spaces with hyphens.
4. Collapse consecutive hyphens.
5. For duplicates, append `-1`, `-2`, etc.

Example: `## 2. Proposed Design` -> `id="2-proposed-design"`

## Risks and limitations
- If an author uses a non-GitHub slug convention in their ToC links, those
  links won't resolve. This is inherent to any auto-slug approach and matches
  how GitHub itself works.
- Buffering heading events adds a small amount of complexity to the event
  loop, but the pattern is straightforward.

## Non-goals
- Auto-generating a table of contents.
- Cross-document anchor links (e.g. `[Link](other.md#section)` — these
  already work since fragments are preserved by link rewriting; the target
  document just needs heading IDs, which this feature provides).
- Visible anchor icons (e.g. hover-to-show `#` link next to headings).
